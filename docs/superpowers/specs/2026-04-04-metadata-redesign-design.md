# メタデータ構造刷新 設計ドキュメント

Date: 2026-04-04

## 背景・目的

現在のメタデータ形式はカラム順序（order）と型情報（attrs）を1つの配列で表現している。
これを `attrs` と `order` に分離し、それぞれ独立して存在できる形式に変更する。

また、`validate($table)` がメタデータなしで返していた動作を変更し、
`attrs`（型推論結果）を含むメタデータを付加して返すようにする。

## 検討したアプローチ

### A（採用）: メタデータ構造の置き換え

- `'#'` の値をハッシュリファレンスに変更し、`attrs`・`order` をサブキーとして持つ
- `validate($table)` が `attrs` のみのメタデータを返すよう変更
- `_attrs` 内部関数を削除し、型推論を `validate` に統合

### B: 新旧両形式をサポート

- 移行期間として旧形式（配列）と新形式（ハッシュ）の両方を読めるようにする
- 不採用理由: このモジュールはまだ外部で使われていないため互換性は不要

### C: メタデータキーを行単位で分ける

- `'#order'` と `'#attrs'` を別行にする
- 不採用理由: `detach`/`attach` が複雑になる

## 採用設計

### 新しいメタデータ形式

```perl
# attrs + order 両方（validate($table, $cols) の戻り値）
{'#' => {
    attrs => {A => 'num', B => 'str', C => 'num'},
    order => ['A', 'B', 'C'],
}}

# attrs のみ（validate($table) の戻り値）
{'#' => {
    attrs => {A => 'num', B => 'str', C => 'num'},
}}

# メタデータなし（変更なし）
[
    {A => 1, B => 'foo', C => 3},
    ...
]
```

### 各関数への影響

| 関数 | 変更内容 |
|---|---|
| `validate($table, $cols)` | `attrs` + `order` 両方を含むメタデータを付加して返す（形式変更） |
| `validate($table)` | **動作変更**: `attrs` のみのメタデータを付加して返す（従来はメタデータなし） |
| `_attrs`（内部） | **削除**: 型推論ロジックは `validate` 内部に吸収 |
| `group` | メタデータなし時に `validate($table)` を呼んで `attrs` を補完し、メタデータ付きで返す |
| `expand` | メタデータなし時に `validate($table)` を呼んで `attrs` を補完し、メタデータ付きで返す |
| `detach` / `attach` | `'#'` の値がハッシュリファレンスになるだけで動作ロジックは変わらず |

### `order` の有無について

- `order` が存在する場合: カラム順序が確定
- `order` が存在しない場合: カラム順序は不定
- `order` なしで `attrs` のみ: `validate($table)` を通した場合（`$cols` 未指定）
- `order` なしで `attrs` なし（メタデータなし）: 依然としてあり得る（手動構築など）

### メタデータ付きテーブルを標準形とする

メタデータ付きテーブルを標準形とする。
メタデータなし AoH が入力された場合でも、必要な処理ではメタデータを補完して扱うため、
戻り値は入力より情報量の多いメタデータ付き形式になることがある。

純粋な AoH が必要な場合は `detach` を利用する。

### 空テーブルの扱い

空テーブル（データ行が 0 件の AoH）はメタデータ付与の対象外とする。

`validate($table)` および `validate($table, $cols)` は、空テーブルに対しては
`attrs`・`order` のいずれも生成せず、メタデータのみを持つ形式も作らない。
空テーブルは常に空の AoH（`[]`）のまま返す。

### 実装方針

各関数は、入口でまずメタデータの有無を判定する。
メタデータ付き入力であれば `detach` によりメタデータとデータ本体を先頭で分離し、
以降の本体ロジックは純粋な AoH に対してのみ行う。

`attrs` の補完や型推論が必要な場合も、メタデータを含む入力全体に対してではなく、
分離後または処理後の純粋な AoH に対して実行する。

処理完了後、必要な場合にのみメタデータを再付与して返す。
再帰処理や中間処理の内部では、メタデータ付き構造をそのまま本体ロジックに渡さない。

### `attrs` と `order` の生成方針

`attrs` と `order` は内部表現として分離して保持する。

通常の生成経路では、`order` は `validate($table, $cols)` によって `attrs` と併せて生成されるため、
`order` のみを持ち `attrs` を持たないメタデータは通常運用では発生しない想定とする。

ただし、手動構築などにより `order` のみを持つメタデータが与えられた場合でも、
`attrs` が必要な処理では型推論を再実行して `attrs` を補って扱う。
