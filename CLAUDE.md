# 新規プロジェクト作成手順

1. ブレインストーミングでパッケージの目的・仕様を検討する
2. `/Users/darong/PRJDEV/<ProjectName>/` にディレクトリ構成を作成する
   - `src/`、`test/`、`docs/`、`.claude/worktrees/`
   - `src/<ProjectName>.pm`（package 宣言のみ）
   - `test/<projectname>.t`（空のテスト）
   - `docs/spec.md`、`docs/test-spec.md`（タイトルのみ）
   - `README.md`（タイトルのみ）
   - `.gitignore`（`.claude/worktrees/` と `CLAUDE.md` を除外）
   - `CLAUDE.md`・`.claude/settings.json`（既存プロジェクトからコピー）
3. 新しいプロジェクトディレクトリに移動してから作業を続ける
   - VS Code の場合：`ファイル` → `フォルダーを開く` で該当ディレクトリを選択
   - ターミナルの場合：`cd /Users/darong/PRJDEV/<ProjectName>`
   - Claude Code の場合：新しいセッションを該当ディレクトリで起動
4. `git init` する
5. 内容を確認してから初回コミットする
6. GitHub にリポジトリを作成してプッシュする

# 変更フロー

1. ブレインストーミングで仕様を検討する
2. `.claude/worktrees/` に worktree を作成してブランチを切る
3. **ブレインストーミング完了直後・実装開始前に** docs/spec.md・docs/test-spec.md を作成・更新してコミットする
4. テストを先に書いて失敗を確認する（TDD）
5. コードを実装してテストが通ることを確認する
6. 4〜5 を機能単位で繰り返す（1テスト1コミットではない）
7. まとまりのある機能が完成したらコミットする
8. README.md を更新する
9. マージ前チェックを実施する（後述）
10. main にマージしたらプッシュを促す
11. プッシュする

# 実装前の確認

- 変更を加える前に変更箇所を提示し、承認を得てから実施する
- 変更規模が大きい場合は特に、差分の概要を先に示す

# ドキュメント同期

- コード変更時は docs/spec.md・docs/test-spec.md・README.md を必ず追随して更新する
- ドキュメントとコードの内容が乖離しないようにする

# マージ前チェック

マージする前に以下を必ず確認する。未対応のものがあれば先に修正してからマージする。

1. **docs/spec.md** が実装内容と一致しているか（仕様変更・追加が反映されているか）
2. **docs/test-spec.md** がテスト内容と一致しているか（テストケースが反映されているか）
3. **README.md** が現在の機能・使い方と一致しているか
4. **全テストが通っているか**

# コミット前の確認

- 変更後は自動でコミットしない。ユーザーが VS Code などでレビューできるよう、変更した旨を伝えて待つ
- 「コミットしてください」と明示的に指示されてからコミットする
- 「コミットしますか？」と聞かない。ユーザーが判断して指示する

# コミット前の README 確認

1. README.md と各種仕様・実装内容に矛盾がないか確認する
2. 追加・更新すべき内容が README.md に反映されているか確認する
3. 不整合や不足があれば修正してからコミットする

# CLAUDE.md の管理方針

- プロジェクトの `CLAUDE.md` は git 対象外とする（`.gitignore` に `CLAUDE.md` を記載）
- プロジェクトの `CLAUDE.md` はこのファイル（`~/.claude/CLAUDE.md`）と同じ内容を基本とし、プロジェクト固有のルール（言語・フレームワーク固有のコーディング規約など）を追記する
- このファイルを変更したら、プロジェクトの `CLAUDE.md` にも同様に反映する
- プロジェクトの `CLAUDE.md` を変更したら、共通ルール部分はこのファイルにも同様に反映する

# Perl コーディング規約（Perl プロジェクトの場合）

- 変数名は小文字基本、役割を直接表す名前を使う（`ref` ではなく `alias`、`part` ではなく `dsl`）
- 複数要素は複数形（`@dsls`、`@rows`、`@cols`）
- 仕様上の用語をそのまま使う（`alias`、`select`、`where`、`having`）
- 短さより文脈との一致を優先する
- 過度な抽象化・設計パターンを持ち込まない。単純なフィルタは `grep`、変換は `map` で完結させる
- `ref`、`part`、`data`、`item`、`obj` のような汎用語は避ける
- 型情報を名前に過剰に埋め込まない（`alias_ref` ではなく `alias`）
- `output_columns`、`working_table` のような長い説明調の名前は避ける（`cols`、`tbl` で十分）
- Perl に合わない設計パターンを持ち込まない。読んですぐ分かることを重視する

# Git 運用

- 変更の種類・規模を問わず、作業は必ず `.claude/worktrees/` 内に worktree を作成して行う
- main のファイルを直接編集しない（CLAUDE.md は例外）
- ブランチ名は `feature/<機能名>` または `fix/<修正内容>` の形式とする
- `git add` はファイル個別指定ではなくプロジェクトディレクトリ単位で行う
- `git status` で未追跡ファイルがないことを確認してからコミットする
